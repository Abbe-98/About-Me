SQL injection lab (client‑side simulation)

Understand the difference between a vulnerable and a hardened solution.

This page simulates how a web application can respond to different types of input. Everything runs in your browser – no data is sent to any server, and the full source code is available on GitHub for transparency and security review.

🧪 Client‑side only 🔍 Educational purpose 📂 Open source 🛡️ No real database

Choose a scenario

Test different input cases and compare what can happen in an insecure implementation versus a secure one.

Normal login

A normal user enters standard credentials. Both implementations should handle this case, but the secure solution does it without building the query directly from raw input.

⚠️ Insecure pattern

The application builds the SQL query directly from text coming from the form.

SELECT * FROM users
WHERE username = [raw input]
AND password = [raw input];

✅ Secure pattern

The application sends input as separate parameters and does not interpret it as query logic.

SELECT * FROM users
WHERE username = ?
AND password_hash = ?;
Strong protection is built in layers: parameterized queries, input validation, hashed passwords, clear error handling, and logging of abnormal patterns.

Analysis result

In this scenario, the risk level is low if the implementation already uses parameterized queries. The biggest difference is that an insecure implementation creates unnecessary future risk as the codebase grows.

Insecure implementation: Risk of future errors
Secure implementation: Input is treated as data

The comparison shows that the secure solution behaves predictably in all three cases, while the insecure one can misinterpret input and create vulnerabilities when logic is built directly into the query string.
